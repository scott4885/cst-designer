# CST Designer Phase 3 Sprint Plan
**Version 1.0 — 2026-04-21**
**Scope:** 3 sprints, ~1 week each
**Supersedes:** all prior ROADMAP docs

## 0. Summary

Phase 3 is the execution block between the approved Bible + PRD V4 and the Phase 4/5 QA and deploy gates. Three streams run in parallel across three one-week sprints, each with a demoable outcome. End-state: a CST Designer that solves on the doctor X-segment graph, renders 10-min blocks legibly, and reproduces the six real SMILE NM / Smile Cascade templates byte-for-byte under the Farran 75%-by-noon default.

**Sprint 1 — Engine + Data.** X-segment data model in Prisma, migration 016, Multi-Column Coordinator MVP enforcing `R-3.1..R-3.6`, four selectable production policies. Stream A leads; B and C prep. Demo: 2-op restorative template generated by the new coordinator with the X-segment audit log.

**Sprint 2 — Grid + Polish.** `ScheduleGridV2.tsx` renders each block as one multi-row grid item with three-zone colouring, sticky headers, hover tooltips, 75/100/125/150 zoom, distinct doctor-exam glyph in hygiene columns. Design tokens + Tailwind 4 config land. Demo: `1D2O_3H` Cascade day at 100% zoom with all twelve UX-L requirements met.

**Sprint 3 — Integration + QA.** Anti-pattern guard goes live. Six real templates reproduced as goldens. Policy selection wired to the derivation popover. `doctorXScheduleJson` round-trips through Excel. Playwright UC-1..UC-10. Coolify deploy behind `CST_V4_ENABLED`. Demo: live UC-1 onboarding, Jameson 50%, Week A+C generation, Excel export, clean inspector.

Phase 3 closes every P0-RES and P1-RES item and every `UX-L1..UX-L12` legibility contract.

## 1. Sprint Master (orchestration)

### 1.1 Cadence

One-week sprints Monday 09:00 to Friday 17:00. Mid-week Wednesday 13:00 is the interface-freeze: Stream A publishes any type-shape deltas before Stream B consumes them Thursday–Friday. Tempo is driven by what the agent team can finish in one coherent pass; tasks that can't land by Wednesday freeze are cut or deferred, not crammed.

### 1.2 Ceremonies

Stand-up is asynchronous — a three-line journal entry in `.cst-rebuild-v3/logs/daily-<sprint>-<day>.md` (yesterday / today / blockers). Any blocker surfaces by noon in the Discord thread with a named owner.

Sprint planning Fridays 15:00–17:00: Tech Lead walks the board, PM validates AC against Bible, Decision Architect ratifies open decisions, Scrum Master commits the cut-line. Board frozen at 17:00 — only defects or user-course-corrections reopen it.

Sprint review Fridays 16:00 — live demo against exit criteria, no slides, running tool only, every demo item passes the legibility and coordinator contracts per Playwright screenshot suite. Retro follows 30 min; output to `retro-<sprint>.md`.

### 1.3 Definition of Ready

A story may enter a sprint only when every one of the following is true:

- Traces to at least one FR in [PRD-V4.md §5–§8](../synthesis/PRD-V4.md) and at least one Bible section in [scheduling-bible.md](../synthesis/scheduling-bible.md).
- Acceptance criteria are expressed as given/when/then or bullet assertions and are machine-checkable.
- Effort is T-shirt sized (XS / S / M / L / XL) and the Tech Lead has signed off.
- Dependencies on other stories or streams are enumerated and those dependencies are either already delivered or scheduled earlier in the same sprint.
- No open question in PRD-V4 §16 or Bible §12 blocks implementation, OR the story explicitly names the safe default to ship absent a stakeholder decision.
- A golden or Playwright test is either pre-existing or stubbed in the same PR.

### 1.4 Definition of Done

Universal — every story in every sprint:

- Code lands on `main` with a PR that green-lights `npm run lint && npm run test && npm run build`.
- Unit tests exist for the new module. For Stream A modules, statement coverage ≥ 85%; for the Multi-Column Coordinator specifically ≥ 95% per NFR-6.
- Golden fixtures for the six real templates still produce identical slot arrays under the Farran 75%-by-noon default policy, OR the delta is reviewed and re-baselined via `UPDATE_GOLDENS=1` with a written rationale in the PR body.
- The story is demoed live at the Friday review from a clean `npm run dev` on a 1440 px viewport in Chrome latest.
- The PR body includes a `Bible: §X.Y` tag and a `Traces to: FR-N` tag; no silent work.
- Deployed behind a feature flag to the Coolify staging (`cst.142.93.182.236.sslip.io/`) no later than the end of Sprint 3.
- Documentation: the relevant section of PRD-V4 or the Bible is cross-linked from the code header comment, and any new public interface is added to `interfaces.md` in `.cst-rebuild-v3/synthesis/`.

## 2. Tech Lead (architecture + task breakdown)

### 2.1 Architectural overview

The Bible prescribes a clean separation the current engine lacks — today `generator.ts` and `slot-helpers.ts` mix pattern lookups, avoid-D-minutes heuristics, and UI-adjacent concerns. Phase 3 splits that mass into five modules:

- **X-Segment Solver** (`src/lib/engine/x-segment/solver.ts`, new) — pure, stateless. Consumes `{provider, blockType, candidateWindow}` and returns `AppointmentSegments` satisfying Bible §2.1 invariants. Replaces scattered segment arithmetic.
- **Multi-Column Coordinator** (`src/lib/engine/coordinator/multi-column.ts`, new) — owns the per-doctor X-segment graph. Enforces `R-3.1..R-3.6`, emits `coordinatorFallbacks[]`. Replaces [rangesAvoidingDMinutes at src/lib/engine/slot-helpers.ts:392](src/lib/engine/slot-helpers.ts#L392).
- **Production Policy Selector** (`src/lib/engine/policy/selector.ts`, new) — four composable policies. Returns `PolicyResult` at save- and generation-time. Replaces the hard-coded 75% in `morning-load-enforcer.ts`.
- **Anti-Pattern Guard** (`src/lib/engine/guard/anti-patterns.ts`, new) — fifteen pure functions (Bible §9). Runs post-generation and at canvas save-time.
- **Pattern Catalog V2** (`src/lib/engine/pattern-catalog-v2.ts`) — demotes [pattern-catalog.ts:19](src/lib/engine/pattern-catalog.ts#L19) to a seed-only fallback. Coordinator is authoritative whenever `maxConcurrentDoctorOps > 1`.

Data layer maps directly from Bible §2. Migration `016_x_segment_model.sql` adds `asstPreMin`, `doctorXMin`, `asstPostMin`, `doctorContinuityRequired` on BlockType; `practiceModel`, `productionPolicy` on Office; `dayOfWeekRoster` on Provider; creates `ProcedureOverride`. Legacy `durationMin/Max` stay as computed columns during V3→V4 coexistence.

### 2.5 Technical risk register

- **Risk A — X-segment backfill heuristic miscalibrates legacy BlockTypes.** Mitigation: T-101 ships dry-run first; a human reviews the backfill report before any production migration. Rollback is a single ALTER TABLE to null-out the new columns.
- **Risk B — Coordinator produces different slot arrays than V3 for the six goldens.** Mitigation: T-109 re-captures goldens after coordinator integration; any shift is reviewed and re-baselined with an explicit PR rationale rather than silently tolerated.
- **Risk C — `ScheduleGridV2.tsx` row-virtualisation breaks sticky headers on Firefox.** Mitigation: T-201 ships with a Firefox-specific `position: sticky` fallback; Playwright runs on Chromium + Firefox + WebKit.
- **Risk D — The policy-selector composition rules (Bible §4.4) produce spurious warnings when two policies overlap.** Mitigation: policy evaluations emit structured `PolicyResult` objects with `source: PolicyName`; the UI renders each warning once, keyed by rule-code and day.
- **Risk E — Hygiene exam window resolution (T-302) conflicts with zigzag offsets (R-3.4) in dense 1D3H days.** Mitigation: T-302 ships with at least three Smile-Cascade fixture days; if the solver cannot satisfy, it emits `coordinatorFallbacks[]` and the UI raises `UX-L9` rather than silently relaxing.
- **Risk F — Migration 016 races with a running Coolify container.** Mitigation: the migration is gated behind maintenance mode; DevOps (T-309) runs it in a staging environment first.
- **Risk G — The 31-task count exceeds single-developer + agent bandwidth in 3 weeks.** Mitigation: T-213 is the explicit cut; T-205 (zoom) can fall back to binary 100%/150% if needed; T-304 inspector panel can ship read-only JSON pre-visualisation.
- **Risk H — ProcedureOverride CRUD leaks into the generator's hot path.** Mitigation: overrides are resolved once per generation at the top of `generateSchedule()` and passed as immutable context; no repeated DB reads inside the loop.

### 2.6 Interface contracts

Stream A exposes and Stream B consumes:

```typescript
// src/lib/engine/types.ts — published at Sprint 1 Wednesday freeze

export interface AppointmentSegments {
  readonly pattern: ReadonlyArray<StaffingCode>;
  readonly durationSlots: number;
  readonly asstPreSlots: number;
  readonly doctorSlots: number;
  readonly asstPostSlots: number;
  readonly resolvedBy: 'COORDINATOR' | 'SEED_FALLBACK';
}

export type StaffingCode = 'A' | 'D' | 'H' | 'E' | null;

export interface PlacedBlock {
  readonly blockId: string;
  readonly providerId: string;
  readonly operatoryId: string;
  readonly startMinute: number;
  readonly segments: AppointmentSegments;
  readonly productionUsd: number;
  readonly tier: 'ROCK' | 'SAND' | 'WATER';
  readonly procedureCategory: ProcedureCategory;
  readonly continuityRequired: boolean;
}

export interface GenerationResult {
  readonly dayOfWeek: Weekday;
  readonly blocks: ReadonlyArray<PlacedBlock>;
  readonly doctorXScheduleJson: Record<string, ReadonlyArray<DoctorXSegment>>;
  readonly warnings: ReadonlyArray<Warning>;
  readonly coordinatorFallbacks: ReadonlyArray<CoordinatorFallback>;
  readonly policyResults: ReadonlyArray<PolicyResult>;
  readonly projectedDoctorProductionPerHour: number;
}

export interface CoordinatorFallback {
  readonly ruleCode: 'R-3.1' | 'R-3.3' | 'R-3.4' | 'R-3.5a' | 'R-3.6';
  readonly blockId: string;
  readonly minute: number;
  readonly relaxation: 'SEED_FALLBACK' | 'DEFERRED' | 'MANUAL_OVERRIDE';
  readonly explanation: string;
}

export interface PolicyResult {
  readonly policy: ProductionTargetPolicy;
  readonly satisfied: boolean;
  readonly metShareUsd: number;
  readonly requiredShareUsd: number;
  readonly violations: ReadonlyArray<{ rule: string; minute?: number }>;
}
```

Stream B must consume `AppointmentSegments.resolvedBy` when rendering `UX-L9` conflict overlays — any block with `resolvedBy === 'SEED_FALLBACK'` at `maxConcurrentDoctorOps > 1` raises `AP-15` and gets the overlay treatment. Stream B must render `doctorXScheduleJson` in the `UX-20` inspector panel. Stream C consumes no types directly but colours the `tier` field in `PlacedBlock` per the design-token ramp.

The freeze deadline is Sprint 1 Wednesday; Stream B's Thursday–Friday work on `ScheduleGridV2.tsx` assumes these types are immutable from that moment. Any Stream A discovery that requires a type change after freeze becomes a Sprint 2 task, not a retroactive patch.

## 5. Sprint Decision Architect (decisions + trade-offs)

### 5.1 Decision framework

Three tiers.

**Author discretion.** Developer on the task decides and logs in the PR body. Examples: test fixture names, log wording, file layout when there's no reuse.

**Team-needed.** Tech Lead chairs; PM and Sprint Facilitator weigh in. Resolved async in PR review or a 15-min Wednesday slot. Examples: new types in `interfaces.md`, library choices, shared test-fixture shapes.

**User approval.** Scott must sign off. Examples: Bible contract changes, departures from PRD-V4, schema changes not in PRD-V4 §9, architectural rework. Logged in `decisions-<sprint>.md` with proposal, rationale, risk — wait for explicit yes before resuming.

### 5.2 Anticipated decisions and pre-resolved trade-offs

We pre-resolve the following now to keep sprints moving. Each carries a one-line rationale.

- **X-segment representation: three scalar columns vs a nested JSON object.** Pre-resolved as three scalars (`asstPreMin`, `doctorXMin`, `asstPostMin`). Reason: Prisma index-ability, SQL query simplicity, alignment with the canonical literature notation; JSON can be added later as a computed field if nested invariants require it.
- **Coordinator algorithm: first-fit vs best-fit.** Pre-resolved as first-fit with a secondary sort on minimum-stagger-displacement. Reason: determinism (NFR-2 requires byte-identical output given a seed), simplicity, alignment with Bible §3.4 zigzag canonical pattern.
- **Legacy `durationMin/Max` retention.** Pre-resolved to keep as computed columns until after Sprint 3 golden-test verification; drop in a Phase 6 follow-up. Reason: rollback safety for the migration.
- **Policy composition default.** Pre-resolved to single-policy-per-office with composability turned off in Sprint 1. Reason: scope; Bible §4.4 explicitly says policies are composable but that complexity pushes to Phase 4.
- **Zoom levels: 4 discrete vs continuous.** Pre-resolved as 4 discrete (75/100/125/150). Reason: row-virtualisation math is cleaner at fixed heights and visual-regression fixtures are simpler.
- **Default production policy for new offices.** Pre-resolved as `FARRAN_75_BY_NOON` (matches current V3 behaviour). Reason: avoids forcing a migration-time policy change on every existing office; user may override per office at any time. This is a change from Bible §4.4's `JAMESON_50_PRIMARY` cold-start default and is explicitly the backwards-compat choice — we flag it as reconcilable at Phase 6.
- **Per-practice procedure override storage.** Pre-resolved in a dedicated `ProcedureOverride` table (PRD-V4 §9.1) rather than a JSON blob on `Office`. Reason: queryability, referential integrity, easier UI CRUD.
- **Anti-pattern guard: fail-closed vs warn.** Pre-resolved as warn-only for Sprint 3 ship. `AP-1` through `AP-15` all raise red banners but do not block save. Reason: user-reported flow interruption risk; fail-closed enforcement moves to Phase 6 once the warning rate on real templates is measured.
- **Feature-flag name.** Pre-resolved as `CST_V4_ENABLED`. Reason: readability, alignment with the V3→V4 boundary.
- **Crown-prep ship default.** Pre-resolved as 80 min (Bible §5.1) with a first-login onboarding prompt offering the efficient 30-min override. Reason: matches SMILE NM and Smile Cascade observed data.
- **Dropping micro-interactions (T-213) if Sprint 2 slips.** Pre-resolved. Reason: they do not appear in any PRD-V4 acceptance criterion or Bible contract.

### 5.3 Open decisions requiring user input

These cannot be pre-resolved and are routed to Scott for a yes/no before the dependent task enters its sprint.
